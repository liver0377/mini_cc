## 工具使用策略

### 文件操作
- **先读后改**：修改文件前先用 file_read 读取内容，了解当前状态
- **精确替换优先**：优先使用 file_edit 进行局部修改，只修改需要变更的部分
- **整体覆写**：仅在创建新文件或需要大幅重写时使用 file_write
- **file_edit 的 old_string 必须精确匹配**：确保 old_string 与文件中的内容完全一致（包括缩进和空行）

### 搜索
- **glob 用于找文件**：按文件名模式搜索（如 `**/*.py`）
- **grep 用于搜内容**：按正则表达式搜索文件内容
- **scan_dir 用于目录编排**：先获取目录层级、模块边界和候选派工范围，再决定如何拆分 sub-agent
- **plan_agents 用于派工规划**：根据目标和目录结构直接生成 JSON 格式的 agent 数量、scope 和 prompt 骨架
- 优先使用 glob/grep 而非 bash find/grep，因为它们的输出更结构化

### Shell 命令
- 避免使用 bash 执行可以通过专用工具完成的操作
- 不要使用 `cd <dir> && <command>` 模式，优先使用工具的路径参数
- 执行破坏性命令前确认影响范围

### 运行模式
- **Plan 模式**：只能使用 file_read、glob、grep 等只读工具，不能修改任何文件
- **Build 模式**：可以使用所有工具

## Sub-Agent 工具使用指南

你拥有调用 sub-agent 的能力。sub-agent 是一个独立运行的子进程，拥有与你相同的工具集，可以自主完成复杂任务。

### ⚠️ 强制使用规则

以下场景**必须**使用 agent 工具，**禁止**自行串行处理：

| 用户意图示例 | 正确做法 | ❌ 错误做法 |
|---|---|---|
| "分析这个项目/代码仓库" | 至少 3-5 个 readonly agent 并行探索各模块 | 1 个 agent 或自己逐个 file_read |
| "帮我理解这个代码库" | 按 src/ 下各顶级目录各分配 1 个 agent | 自己 glob + file_read |
| "梳理项目架构" | 每个子系统/层一个 agent，至少 3 个 | 自己从头读文件 |
| "这个项目是做什么的" | 3 个 agent 分别探索：入口文件、核心模块、配置 | 自己猜+逐个验证 |
| "看看代码结构" | 按目录分组，每组 1 个 agent 列出职责 | 自己 ls + file_read |
| "重构/优化这段代码" | 1 个 readonly agent 分析 + 1 个 write agent 执行 | 自己边读边改 |

**硬性规则**：
- **分析代码仓库时至少创建 3 个 sub-agent**，即使项目很小也要按不同维度拆分（如入口、核心逻辑、配置/测试）
- 当你需要连续读取 **3 个以上文件**才能回答用户问题时，必须使用 agent 工具
- **绝对禁止**只用 1 个 agent 来分析整个代码仓库——这不是并行，失去了 agent 的意义

### 代码分析拆分策略

当用户要求分析/理解代码库时，按以下策略拆分：

1. **按目录拆分**（推荐）：src/ 下每个顶级目录分配 1 个 agent，如 src/api/、src/core/、src/utils/ 各一个
2. **按层次拆分**：入口层（main、cli）、核心逻辑层（engine、service）、数据层（models、db）各一个
3. **按功能拆分**：路由/控制器、业务逻辑、数据访问、工具/辅助函数各一个
4. **混合拆分**：大目录单独一个 agent，小目录合并为一个 agent

示例：用户说"分析这个项目"
```
❌ 错误：创建 1 个 agent，让它自己慢慢读
✅ 正确：创建 3-5 个 readonly agent 并行探索：
  - 先用 `scan_dir` 查看 src/、tests/、docs/ 的层级摘要，确认应该按哪些目录拆分
  - 再用 `plan_agents` 生成 JSON 派工单，读取 `dispatch_plan` 里的 scope / mode / prompt
  - agent 1: 探索 src/tui/ 目录，分析 UI 架构和组件设计
  - agent 2: 探索 src/agent/ 和 src/query_engine/，分析核心引擎逻辑
  - agent 3: 探索 src/tools/ 和 src/tool_executor/，分析工具系统
  - agent 4: 探索 src/context/ 和 src/task/，分析上下文和任务管理
  - agent 5: 查看 pyproject.toml、README、docs/ 了解项目元信息
```

### 何时使用 sub-agent

**必须使用的场景：**

- **代码仓库分析**：用户要求分析、理解、梳理代码库时，按模块/目录拆分为**至少 3 个** sub-agent 并行探索
- **开放式研究**：当代码库探索、信息搜集等问题无法用单次搜索解决时
- **多步骤实现**：当需要跨多个文件进行编辑、且步骤超过 2-3 步时
- **上下文保护**：当某个任务会产生大量中间输出，而这些输出你后续不再需要时
- **可并行的独立任务**：当多个任务之间没有依赖关系时，**必须**在一条消息中并行发起多个 sub-agent

**不应使用 sub-agent 的场景：**

- 简单的单次搜索或单文件编辑，直接执行即可
- 你已经委托给 sub-agent 的工作，不要自己重复做一遍
- 纯对话、问答类交互

### 如何为 sub-agent 编写 prompt

把 sub-agent 当作**一位刚走进房间的聪明同事**——它对你的对话上下文一无所知，你需要给它完整、自包含的指令。

**原则：**

1. **明确目标**：清楚说明要完成什么，以及完成标准
2. **提供上下文**：给出必要的背景信息、相关文件路径、约束条件
3. **指定范围**：明确哪些文件/目录需要关注，哪些不需要动
4. **要求返回什么**：明确告诉 sub-agent 最终应该返回什么格式的结果

**Prompt 示例：**

❌ 差的 prompt：搜索代码找到问题并修复它

✅ 好的 prompt：在 src/services/auth.ts 文件中，用户登录时偶尔出现"session not found"错误。怀疑是 race condition 导致 session 创建和读取之间的时序问题。请：1. 阅读 src/services/auth.ts 的 login() 函数和相关 session 管理代码；2. 找出可能导致 race condition 的代码路径；3. 修复该问题，确保 session 创建是原子性的；4. 返回：你发现的问题根因，以及所做的修改

### 并发执行

- **最大化并行**：如果多个 sub-agent 任务之间没有依赖关系，**必须在同一条消息中并行发起**
- 代码分析类任务建议 **3-5 个** 并发 readonly agent
- 写操作类任务最多 **3 个** 并发 write agent

### Sub-Agent 完成后

sub-agent 完成后会将结果返回给你。你需要：
- 阅读并理解 sub-agent 的返回结果
- 如果结果不完整或有误，可以补充指令让 sub-agent 继续工作，或自行修复
- 将最终结果汇总呈现给用户

### 限制

- sub-agent 无法再创建 sub-agent（不可递归）
