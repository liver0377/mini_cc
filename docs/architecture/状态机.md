# 状态机

本文总结当前系统内部的主要状态流转。系统里有几条相互关联的状态线：交互式 Query loop、Harness Run、Step、WorkItem、SubAgent 和 Task。整体原则是：上层状态由下层执行结果驱动，下层执行结果通过事件、`StepResult`、生命周期记录和持久化状态回写到上层。

## 总览

```
用户输入
   │
   ├── 交互式路径
   │      ▼
   │   QueryEngine
   │      ├── LLM stream
   │      ├── Tool calls
   │      ├── Compaction
   │      └── QueryState 更新
   │
   └── Harness 路径
          ▼
       RunHarness
          ▼
       SupervisorLoop
          ├── Scheduler 选择 WorkItem
          ├── StepRunner 执行 Query / Bash / Agent
          ├── RunJudge 评估健康度
          ├── PolicyEngine 决定状态迁移
          └── CheckpointStore 持久化状态
```

## RunState 状态机

`RunState.status` 描述一次 Harness run 的整体生命周期。

### 状态集合

```
CREATED        # 已创建，尚未启动
PLANNING       # 预留状态，当前主循环中较少直接使用
RUNNING        # 正在执行
COOLDOWN       # provider 瞬时失败后的退避等待
VERIFYING      # 正在运行测试类步骤
BLOCKED        # 兼容历史状态，当前策略不再主动落到该状态
WAITING_HUMAN  # 预留状态，当前策略不再主动落到该状态
COMPLETED      # 成功完成
FAILED         # 失败终止
CANCELLED      # 用户取消
TIMED_OUT      # 超时终止
```

当前代码里 `RunState.is_terminal` 只包含：

```
BLOCKED
COMPLETED
FAILED
CANCELLED
TIMED_OUT
```

其中 `BLOCKED` 保留为历史兼容终态；当前 `PolicyEngine` 遇到保护性阻断场景时会返回 `PolicyAction.BLOCK`，但最终 `terminal_status` 会落到 `FAILED`。

### 主状态迁移

```
CREATED
   │ SupervisorLoop.run_with_interrupt()
   ▼
RUNNING
   │
   ├── 用户取消
   │      ▼
   │   CANCELLED
   │
   ├── 超过 deadline
   │      ▼
   │   TIMED_OUT
   │
   ├── 超过测试 / Bash / Agent 并发预算
   │      ▼
   │   FAILED
   │
   ├── provider transient failure
   │      ▼
   │   COOLDOWN
   │      │ cooldown_until 到期
   │      ▼
   │   RUNNING
   │
   ├── RUN_TESTS 执行中
   │      ▼
   │   VERIFYING
   │      │ step 处理完成
   │      ▼
   │   RUNNING / terminal
   │
   ├── FINALIZE 成功
   │      ▼
   │   COMPLETED
   │
   └── 没有 pending Step 且没有 active Agent
          ▼
       COMPLETED
```

## SupervisorLoop 循环

`SupervisorLoop` 每轮循环都围绕 `RunState` 做一次调度、执行和决策。

```
while not run_state.is_terminal:
    1. 检查 interrupt_event
    2. 如果 cooldown 未到期，保持 COOLDOWN 并等待
    3. 如果 cooldown 到期，恢复 RUNNING
    4. drain SubAgent 生命周期事件，刷新 spawned_agents
    5. drain 已完成的后台只读 WorkItem
    6. 检查 run 级预算和并发限制
    7. 优先批量派发 readonly WorkItem
    8. Scheduler 选择下一个 WorkItem
    9. 标记 Step / WorkItem 为 IN_PROGRESS
   10. StepRunner 执行具体工作
   11. 保存 artifacts、trace spans、iteration snapshot/review
   12. RunJudge 评估健康度
   13. PolicyEngine 生成决策
   14. 应用决策并持久化 checkpoint

退出循环后：
    1. 刷新 Agent 指标
    2. 生成 Documentation.md
    3. 保存最终 RunState
```

## Step 状态机

`Step.status` 描述一个高层阶段的聚合执行状态。所有真实可执行单元都会被封装成 `WorkItem`；如果外部传入的 Step 没有 WorkItem，系统会自动补一个默认的 `step-id.main` WorkItem。

### 状态集合

```
PENDING
IN_PROGRESS
SUCCEEDED
FAILED_RETRYABLE
FAILED_TERMINAL
SKIPPED
```

### 状态迁移

```
PENDING
   │ 其下某个 WorkItem 被 Scheduler 选中
   ▼
IN_PROGRESS
   │
   ├── PolicyAction.CONTINUE
   │      ▼
   │   SUCCEEDED
   │
   ├── PolicyAction.COMPLETE
   │      ▼
   │   SUCCEEDED + RunState.COMPLETED
   │
   ├── PolicyAction.RETRY
   │      ▼
   │   PENDING（retry_count + 1）
   │
   ├── PolicyAction.COOLDOWN
   │      ▼
   │   PENDING（retry_count + 1）+ RunState.COOLDOWN
   │
   ├── PolicyAction.REPLAN
   │      ├── result.success == true  ──► SUCCEEDED
   │      └── result.success == false ──► FAILED_RETRYABLE
   │
   ├── PolicyAction.BLOCK
   │      ▼
   │   FAILED_TERMINAL + RunState.FAILED
   │
   └── PolicyAction.FAIL
          ▼
       FAILED_TERMINAL + RunState.FAILED
```

`StepKind.FINALIZE` 是特殊步骤类型。它不是状态，但它成功后 `PolicyEngine` 会返回 `COMPLETE`，从而把 run 推到 `COMPLETED`。

## WorkItem 状态机

`WorkItem` 是 Scheduler 的统一调度单位。一个 Step 至少包含一个 WorkItem，也可以包含多个 WorkItem。

### 状态集合

```
PENDING
IN_PROGRESS
SUCCEEDED
FAILED_RETRYABLE
FAILED_TERMINAL
SKIPPED
```

### 普通 WorkItem 迁移

```
PENDING
   │ 依赖 WorkItem 全部 SUCCEEDED，Scheduler 选中
   ▼
IN_PROGRESS
   │
   ├── StepResult.success == true
   │      ▼
   │   SUCCEEDED
   │
   ├── StepResult.retryable == true 且 retry_count 未超限
   │      ▼
   │   PENDING（retry_count + 1）
   │
   └── 不可重试或重试耗尽
          ▼
       FAILED_TERMINAL
```

`SupervisorLoop` 会在单个 WorkItem 成功后继续调度剩余 WorkItem。只有当所有 WorkItem 都完成，或者出现终止性失败时，才会最终收束到 Step 级决策。

### 后台只读 WorkItem 迁移

只读角色包括：

```
analyzer
planner
reporter
verifier
```

这些 WorkItem 可以被批量派发给后台 SubAgent。

```
PENDING
   │ Scheduler.decide_readonly_batch()
   ▼
IN_PROGRESS
   │ start_readonly_work_item_background()
   ▼
后台 SubAgent 运行中
   │
   ├── Agent completed(success=true)
   │      ▼
   │   WorkItem.SUCCEEDED
   │
   └── Agent completed(success=false) / cancelled
          ▼
       WorkItem.FAILED_TERMINAL
```

后台结果不是立即返回给当前执行栈，而是在后续循环中由 `_drain_completed_background_work_items()` 读取 `AgentTrace` 和输出文件，再回写到 WorkItem、Step 和 RunState。

## SubAgent 状态机

`AgentStatus` 描述 runtime 层 SubAgent 的生命周期。

### 状态集合

```
CREATED
RUNNING
BACKGROUND_RUNNING
COMPLETED
CANCELLED
```

### 前台 Agent

```
CREATED
   │ run_agent()
   ▼
RUNNING
   │
   ├── QueryEngine 正常结束
   │      ▼
   │   COMPLETED
   │
   └── cancel_agents()
          ▼
       CANCELLED
```

### 后台 Agent

```
CREATED
   │ start_background_agent()
   ▼
BACKGROUND_RUNNING
   │
   ├── run_background() 正常结束
   │      ▼
   │   COMPLETED
   │
   └── cancel_agents()
          ▼
       CANCELLED
```

SubAgent 的生命周期事件会被 `SupervisorLoop` drain，转换成 `RunState.spawned_agents` 里的 `AgentTrace`。`AgentTrace` 不只是状态记录，也会保存 readonly 标记、scope、输出路径、stale 标记和终止原因。

## QueryEngine 状态流

`QueryEngine` 没有显式的枚举状态，它的状态主要体现在 `QueryState.messages`、`turn_count` 和事件流中。

```
准备消息
   │
   ▼
LLMProvider.stream()
   │
   ├── TextDelta
   ├── ToolCallStart
   ├── ToolCallDelta
   └── ToolCallEnd
          │
          ▼
收集 ToolCall
   │
   ├── 无 ToolCall
   │      ▼
   │   本轮结束，更新 QueryState
   │
   └── 有 ToolCall
          ▼
       StreamingToolExecutor.execute()
          │
          ├── 安全工具并发执行
          └── 非安全工具串行执行
          │
          ▼
       ToolResultEvent
          │
          ▼
       工具结果追加到消息列表
          │
          ▼
       检查是否需要 compaction
          │
          ▼
       进入下一轮 LLM 调用
```

因此 QueryEngine 的终止条件是：当前 LLM 响应不再产生工具调用，或遇到上下文/执行错误并返回对应事件。

## Task 状态机

`Task` 是 runtime 任务队列里的通用任务模型，当前主要用于本地 Agent 和 Bash 任务。

### 状态集合

```
PENDING
IN_PROGRESS
COMPLETED
FAILED
CANCELLED
```

### 状态迁移

```
PENDING
   │ executor / dispatcher 领取
   ▼
IN_PROGRESS
   │
   ├── 执行成功
   │      ▼
   │   COMPLETED
   │
   ├── 执行失败
   │      ▼
   │   FAILED
   │
   └── 用户取消 / 上层取消
          ▼
       CANCELLED
```

## 状态之间的驱动关系

```
ToolResultEvent / AgentCompletionEvent
        │
        ▼
StepResult
        │
        ▼
WorkItem.status / Step.status
        │
        ▼
RunJudge.assess()
        │
        ▼
PolicyEngine.evaluate_step()
        │
        ▼
RunState.status
```

也就是说，系统不是由单个全局状态机驱动，而是由多层状态机组成：

1. QueryEngine 负责一次对话轮次内的 LLM 与工具循环。
2. SubAgent 负责一个独立 Agent 的生命周期。
3. WorkItem 负责 Step 内部的细粒度调度。
4. Step 负责 Harness 的高层执行阶段。
5. RunState 负责整个 run 的生命周期和终止条件。

`SupervisorLoop` 是这些状态线的汇聚点：它读取下层事件和结果，更新中层 Step/WorkItem，再通过策略决策推进顶层 RunState。
